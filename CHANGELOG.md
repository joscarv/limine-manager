# Changelog

## [1.5.0] - 2026-07-20

### Added

- Secure Boot support for traditional Limine installations.
- Automatic BLAKE2b hashing for kernel and initramfs resources.
- Secure Boot validation.
- EFI signature verification.
- Automatic Limine re-signing.
- Automatic configuration enrollment.
- Transactional rollback for EFI updates.
- Secure Boot configuration section.

### Fixed

- Correct handling of sbctl verification when running without elevated privileges.
- Fixed Secure Boot signing transaction after configuration enrollment.
- Improved EFI backup handling during failed transactions.

### Changed

- Improved validation diagnostics.
- Improved refresh workflow.
- Improved status reporting.
- Improved preview generation.

### Safety

- Installing 1.5.0 does not modify `/boot`.
- The Limine EFI executable is backed up before enrollment.
- Failed enrollment, signing, or verification restores both the EFI executable and previous configuration.

All notable changes to this project are documented here. The project follows Semantic Versioning.

## [1.4.3] - 2026-07-16

### Fixed

- Distinguish all snapshots returned by Snapper from the snapshots selected for the Limine menu.
- Apply `[snapper] max_snapshots` and read-write filtering through one shared selector used by discovery and generation.
- Base the large-menu warning on selected menu snapshots instead of the total Snapper inventory.

### Changed

- `status` now reports `Available snapshots`, `Menu snapshots`, and `Maximum configured`.
- `SystemModel` now carries an explicit snapshot model containing available and selected collections.

### Documentation

- Updated `AGENTS.md`, `docs/DECISIONS.md`, and `docs/PROJECT_CONTEXT.md` for the next development session.

### Tests

- Added regression coverage for a host with 150 available snapshots and a configured menu limit of 10.

## [1.4.2] - 2026-07-16

### Fixed

- Discover dm-crypt backing devices through sysfs so unprivileged `status` runs can still report the backing device, LUKS UUID, and PARTUUID.
- Preserve `cryptsetup status` and `lsblk PKNAME` as fallback discovery methods.

### Tests

- Added a regression scenario for a traditional encrypted installation where `cryptsetup status` fails without root privileges.

## [1.4.1] - 2026-07-16

### Fixed

- Discover the encrypted backing device primarily from `cryptsetup status`, with `lsblk PKNAME` retained as a fallback.
- Populate LUKS UUID and backing PARTUUID reliably for dm-crypt roots.
- Print filesystem paths without `std::filesystem::path` quoting.
- Display the normalized root device separately from its Btrfs subvolume.

### Changed

- `status` distinguishes healthy, degraded, and unhealthy installations.
- Encrypted roots with incomplete backing metadata produce a warning and exit code 2.
- Configuration state is reported as `synchronized` or `changes pending`.
- Snapshot menus larger than 100 entries are highlighted in the report.

## [1.4.0] - 2026-07-16

### Added

- Added a central `SystemModel` containing system, Snapper, and snapshot discovery results.
- Added `DiscoveryService` as the single application entry point for host inspection.
- Expanded `status` into a complete health report covering root storage, encryption, boot, kernels, snapshots, configuration drift, themes, and backups.
- `status` now reports validation problems instead of aborting before useful diagnostics are shown.

### Changed

- Refactored command execution around the explicit Discovery → Model → Validation → Generation → Apply pipeline.
- Refresh now performs each rediscovery through the same centralized discovery service.
- `status` keeps exit code `0` for a healthy and current system, `3` for healthy systems with pending configuration changes, and `1` for unhealthy systems.

## [1.3.3] - 2026-07-16

### Fixed

- Accept `cryptdevice=PARTUUID=<partuuid>:<mapper>` as generated by Archinstall with the traditional `encrypt` mkinitcpio hook.
- Resolve encryption mappings by stable device identity: LUKS UUID, partition PARTUUID, canonical backing-device path, or mapper name.
- Preserve the encrypted backing partition PARTUUID in system discovery instead of validating only against the LUKS UUID.
- Normalize Btrfs mount sources such as `/dev/mapper/root[/@]` before mapper and block-device inspection.
- Added a regression test reproducing the reported Archinstall UKI + encrypted Btrfs + `cryptdevice=PARTUUID=` installation.

## [1.3.2] - 2026-07-16

### Fixed

- Encryption validation now identifies the encrypted root primarily by its LUKS UUID instead of requiring the mapper alias in `cryptdevice=` or `rd.luks.name=` to exactly match the currently mounted `/dev/mapper/*` name.
- `rd.luks.name=<uuid>` without an explicit mapper alias is now accepted when the UUID matches the backing LUKS container.
- Optional `cryptdevice=` fields after the mapper name are handled correctly.
- The UUID of the filesystem inside an encrypted mapper is now detected, allowing `root=UUID=<btrfs-uuid>` configurations generated by Archinstall to validate correctly.
- Added a regression test for an Archinstall-style UKI, encrypted Btrfs root, differing mapper aliases, and `root=UUID=`.

## [1.3.1] - 2026-07-16

### Fixed

- Discover `limine.conf` recursively below the configured boot mount, preferring `EFI/BOOT/limine.conf` when the configured path is absent.
- Discover kernel, initramfs, and CPU microcode images in boot subdirectories instead of requiring them at the root of `/boot`.
- Preserve boot-relative subdirectories when rendering Limine `path` and `module_path` values.
- Detect Arch Linux Unified Kernel Images such as `EFI/Linux/arch-linux.efi` and render them with Limine's `efi` protocol.
- Do not require a standalone initramfs or microcode image when a matching UKI is available.

## [1.3.0] - 2026-07-15

### Added

- Generate a canonical kernel command line when `/etc/kernel/cmdline` is missing or empty.
- Detect direct filesystem UUIDs for unencrypted roots.
- Detect dm-crypt mapper names, backing devices, and LUKS UUIDs for encrypted roots.
- Select `cryptdevice=UUID=...:mapper` or `rd.luks.name=UUID=mapper` from the configured mkinitcpio hooks.
- Add regression tests for unencrypted Btrfs, traditional `encrypt`, and `sd-encrypt` generation.

### Fixed

- Stop requiring LUKS parameters when the mounted root filesystem is not encrypted.
- Accept `root=UUID=...` when it identifies the mounted unencrypted root filesystem.

### Compatibility

- A non-empty `/etc/kernel/cmdline` remains authoritative and is never overwritten.
- Configuration schema remains version 1.

## [1.2.0] - 2026-07-15

### Added

- Add `refresh` as the canonical non-interactive synchronization command.
- Add `request-refresh` for lightweight event sources to coalesce refresh requests under `/run/limine-manager`.
- Add `automation-status` for read-only inspection of automation configuration and installed integration files.
- Add strict schema-1 `[automation]` configuration defaults for Snapper, pacman, debounce, and runtime state.
- Add a Snapper plugin adapter and systemd refresh service/timer installation rules.
- Add tests for Snapper event filtering, ignored unrelated events, pending refresh coalescing, and automation configuration validation.

### Changed

- Change the pacman helper to request the centralized asynchronous refresh path instead of running validate/apply directly.

## [1.1.0] - 2026-07-13

### Added

- Add guarded Btrfs snapshot rollback commands: `rollback-status`, `rollback-plan`, and `rollback`.
- Add rollback planning models, Btrfs infrastructure, execution phases, locking, topology verification, and partial recovery after failed subvolume moves.
- Preserve the previous main root subvolume under a unique `@.limine-manager.rollback.<snapshot>.<transaction>` name before replacing `@`.
- Regenerate `limine.conf` after a successful Btrfs rollback using the existing apply workflow.
- Add regression tests for rollback eligibility, conflicts, successful execution, and recovery after a failed final move.
- Add built-in visual theme presets and a `themes` command to list them.

### Compatibility

- Configuration schema remains version 1.
- Existing `apply`, `restore`, backup, and Limine rendering behavior remain unchanged.
- `restore` still restores only managed `limine.conf` backups; it does not perform Btrfs system rollback.

## [1.0.1] - 2026-07-13

### Fixed

- Normalize Btrfs `findmnt` sources such as `/dev/mapper/cryptroot[/@]` before comparing them with `root=`.
- Support both mkinitcpio encryption parameter families: `cryptdevice=UUID=...:mapper` and repeatable `rd.luks.name=UUID=mapper`.
- Add regression tests for `sd-encrypt`, traditional `encrypt`, repeated `rd.luks.name`, and Btrfs subvolume source suffixes.

## [1.0.0] - 2026-07-13

### Added

- Stable command-line interface for validation, preview, planning, application, backup management, and restore.
- Configuration schema version 1 through `[manager] schema_version = 1`.
- `check-config` command that validates configuration without requiring a compatible host system.
- Bash and Zsh completion files.
- Release checklist, security policy, migration guide, and stable release notes.

### Changed

- Project version and packaging metadata updated to 1.0.0.
- Effective configuration output now includes the schema version.
- Future unsupported configuration schema versions are rejected explicitly.

### Compatibility

- Configuration files created for 0.9.0 remain valid. A missing schema version is interpreted as schema 1.
- Existing commands and exit codes from 0.9.0 remain unchanged.

## [0.9.0] - 2026-07-13

- Added release engineering, CI, sanitizers, coverage, formatting, man page, and reproducible source archives.
